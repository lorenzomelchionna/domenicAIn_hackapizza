Agent Edition — Ciclo Cosmico 790 — Tutto quello che serve per vincere

🎯 Obiettivo

Ogni team gestisce un ristorante galattico nel Multiverso del Ciclo Cosmico 790. L'obiettivo e' massimizzare il saldo del ristorante — simbolo di successo economico e gestionale.

Il Core Loop

Decidere cosa cucinare e per chi → Acquistare ingredienti rari → Interagire e competere con altri ristoratori → Servire clienti alieni esigenti → Guadagnare.

🧪 Ingredienti: Risorsa Scarsa, Variabile e Deperibile

⚠️ REGOLA FONDAMENTALE: Ingredienti Deperibili

Gli ingredienti scadono a fine turno — vittime di instabilita' quantiche, decadimenti biologici o rotte logistiche troppo lunghe.

Questo rende cruciale:

  • Pianificare l'acquisto in base a cio' che si riuscira' a cucinare
  • Evitare di spendere su ingredienti inutilizzati
  • Adattare il menu a cio' che si ha realmente, non a cio' che si desidererebbe avere

Gli ingredienti sono rari, deperibili e spesso contesi. Possederli significa potere temporaneo. Perderli significa opportunita' sprecate.

🍝 Le Ricette

Ogni ricetta racconta una storia — nata su pianeti agricoli, monasteri orbitanti o cucine improvvisate durante guerre stellari.

Le ricette determinano:

  • Ingredienti necessari
  • Tempo di preparazione
  • Prestigio del piatto

Recupera la lista completa via GET /recipes — array ricette con ingredienti e tempi.

📋 Il Menu: Posizionamento Strategico

Il menu comunica a chi ti rivolgi e chi intendi respingere. Prezzi, ricette e scelte alimentari definiscono il tuo ruolo nel Multiverso.

Impostare un menu significa:

  • Dichiarare quali piatti il ristorante e' disposto a servire
  • Stabilire il prezzo di ciascun piatto
  • Determinare il tipo di clientela che frequentera' il ristorante

Strategia di Prezzo

📉 Prezzi Bassi

Attirano clienti sensibili al costo ma richiedono alto volume e grande efficienza.

📈 Prezzi Alti

Puntano su margini migliori, ma rischiano di allontanare chi ha budget limitati.

🚫 Intolleranze e Restrizioni Alimentari

⚠️ ATTENZIONE CRITICA

Nel Multiverso un errore alimentare non e' solo un reclamo. Puo' essere un incidente diplomatico. O peggio.

Ignorare le intolleranze = rischiare vite, reputazione e sanzioni federali.

NON servire piatti che contengano ingredienti a cui il cliente e' intollerante!

🚪 Apertura e Chiusura del Ristorante

Non esiste l'obbligo di restare aperti. Saper quando chiudere e' saggezza gestionale.

✅ Restare Aperti

Permette di accumulare saldo e reputazione.

🔒 Chiudere

Evita il collasso operativo. Un ristorante chiuso o senza menu e' invisibile ai clienti.

Tool: update_restaurant_is_open — In serving phase si puo' solo chiudere (non riaprire).

🏪 Mercato Interno: Scambio tra Ristoranti

Oltre all'asta, i ristoranti possono scambiare tra loro. Il mercato e' pubblico e visibile a tutti.

💡 Sii Reattivo!

Non aspettare che gli altri ristoranti ti rubino gli ingredienti migliori!

Cosa permette:

  • Comprare ingredienti mancanti quando l'approvvigionamento non e' bastato
  • Vendere surplus che altrimenti scadrebbe
  • Costruire alleanze (o trappole) economiche

Le offerte vengono annullate alla fine di ogni turno.

Tool del Mercato

create_market_entry

Crea proposta BUY o SELL. Genera broadcast message a tutti.

execute_transaction

Accetta un'entry di mercato esistente.

delete_market_entry

Rimuove una tua entry dal mercato.

Leggi il mercato: GET /market/entries

🏆 Come Si Vince

Vince chi riesce a:

  • Trasformare ingredienti limitati in piatti serviti con continuita'
  • Leggere bene le richieste e ridurre errori/sprechi
  • Gestire competizione e mercato con tempismo
  • Proteggere il saldo evitando acquisti inutili e ingredienti sprecati
  • Guadagnare di piu' alla fine della sfida!

Ogni errore ha un costo

Ingrediente sprecato = perso nel vuoto cosmico. Cliente servito male o non servito = danno reputazione + non paga la consumazione!

🔄 Ciclo Completo di un Turno

Ogni turno rappresenta una giornata operativa. Durata indicativa: 5-7 minuti (variabile, segui SSE).

📡 SSE: game_started → Inizio nuovo turno
↓
SPEAKING Fase 1: Pianificazione e Diplomazia

Trattative a porte chiuse. Le rotte ufficiali non sono ancora aperte.

  • ✅ send_message — parlare con altri team, negoziare, stringere alleanze
  • ✅ save_menu — impostare/aggiornare il menu
  • ✅ update_restaurant_is_open — aprire/chiudere ristorante
  • ✅ Mercato interno (create/execute/delete_market_entry)
  • ✅ restaurant_info, get_meals — lettura dati
  • ❌ NO asta ingredienti
  • ❌ NO preparazione/servizio
📡 SSE: game_phase_changed → phase: "closed_bid"
↓
CLOSED BID Fase 2: Asta Cieca Ingredienti

Canali della Federazione aperti. Corsa agli ingredienti. Nessuno conosce le offerte degli altri.

  • ✅ closed_bid — inviare offerte (ingredient, bid, quantity)
  • Ingredienti limitati — chi offre di piu' ha priorita'
  • Non garantito ottenere tutto: potresti ricevere solo una parte
  • Se invii piu' volte nel turno: vale ultima submission
  • I prezzi degli altri noti solo a asta conclusa (GET /bid_history)
  • ✅ Ancora: save_menu, mercato, send_message
  • ❌ NO preparazione/servizio
📡 SSE: game_phase_changed → phase: "waiting"
↓
WAITING Fase 3: Assestamento Strategico

Silenzio operativo. Le scelte diventano irrevocabili.

  • ✅ Analizzare ingredienti ottenuti (GET /restaurant/:id)
  • ✅ save_menu — aggiornare menu finale
  • ✅ Mercato interno — comprare ingredienti mancanti, vendere surplus
  • ✅ send_message, update_restaurant_is_open
  • ❌ NO preparazione/servizio
📡 SSE: game_phase_changed → phase: "serving"
↓
SERVING Fase 4: Il Servizio

Le porte si aprono e arrivano i clienti.

📡 SSE: client_spawned → { clientName, orderText } — un cliente arriva!
  • ✅ Interpretare richiesta cliente (verificare intolleranze!)
  • ✅ prepare_dish → avvia preparazione
📡 SSE: preparation_complete → { dish } — piatto pronto!
  • ✅ serve_dish (dish_name, client_id) → serve al cliente
  • ✅ Mercato interno ancora attivo
  • ✅ update_restaurant_is_open → solo chiusura
  • ⚠️ Cliente non servito o servito male = non paga + danno reputazione
  • ❌ NO save_menu

⚠️ Ingredienti Deperibili

A fine turno (stopped) TUTTI gli ingredienti non usati scadono e vengono persi.

📡 SSE: game_phase_changed → phase: "stopped"
↓
STOPPED Fine Turno
  • 🗑️ Ingredienti non usati scadono
  • 💰 Aggiornamento saldo
  • ⭐ Aggiornamento reputazione
  • ✅ Solo lettura (restaurant_info, get_meals)
  • ❌ Tutte le altre operazioni bloccate
↓
Nuovo turno inizia... → torna a game_started

📡 Sistema Eventi SSE (Server-Sent Events)

Connessione

Metodo: GET

Path: /events/:restaurantId

Header: x-api-key

Handshake: data: connected

Regole:

  • Solo una connessione SSE attiva per ristorante
  • Puoi connetterti solo al tuo restaurantId

Errori connessione:

  • 401 — API key non valida
  • 403 — restaurantId non tuo
  • 404 — ristorante inesistente
  • 409 — connessione gia' attiva

📨 Catalogo Eventi

game_started — broadcast a tutti

Payload: {}

Indica l'inizio di un nuovo turno di gioco.

game_phase_changed — broadcast a tutti

Payload: { phase: "speaking" | "closed_bid" | "waiting" | "serving" | "stopped" }

Indica il cambio di fase nel turno corrente. Segui sempre questo evento per triggerare la logica specifica.

client_spawned — solo al ristorante destinatario

Payload: { clientName: string, orderText: string }

Un cliente e' arrivato e ha fatto una richiesta. Solo durante serving phase.

preparation_complete — solo al ristorante destinatario

Payload: { dish: string }

La preparazione del piatto e' completata, pronto per essere servito con serve_dish.

message — broadcast a tutti

Payload: { sender: string, payload: any }

Messaggio pubblico (es. nuova entry di mercato creata via create_market_entry).

new_message — solo al ristorante destinatario

Payload: { messageId, senderId, senderName, text, datetime }

Messaggio privato ricevuto da altro team via send_message.

heartbeat — broadcast a tutti

Payload: { ts: number } (epoch milliseconds)

Health check connessione. Ignorare nella logica di gioco.

game_reset — broadcast a tutti

Payload: {}

Reset completo del gioco (es. tra run). Evento di servizio piattaforma.

🔑 Pattern di Implementazione

  1. Aprire connessione SSE all'avvio dell'agente
  2. Gestire eventi in modo asincrono (non bloccare)
  3. Riconnettere automaticamente se la connessione cade
  4. Ignorare heartbeat e game_reset nella logica di gioco
  5. Su game_phase_changed: triggerare la logica specifica della fase

🌐 HTTP GET Endpoints

Tutti richiedono header x-api-key. Disponibili in tutte le fasi (lettura).

GET /meals

Restituisce le richieste dei clienti con i relativi ID per un dato ristorante in un dato turno.

Query obbligatorie: turn_id, restaurant_id

Risposta: include campo booleano executed (true se pasto gia' servito).

GET /restaurants

Overview di tutti i ristoranti in gioco.

GET /recipes

Array di tutte le ricette disponibili con ingredienti e tempi di preparazione.

GET /bid_history?turn_id=<id>

Tutte le puntate gia' concluse di tutti i team in un dato turno. Utile per analizzare strategie di bidding avversarie.

GET /restaurant/:id

Informazioni dettagliate sul proprio ristorante (saldo, inventario, menu, reputazione).

Vincolo: puoi leggere solo il tuo ristorante.

Errori:

  • 400 — ID non valido
  • 403 — ID non tuo
  • 404 — ristorante non trovato
GET /restaurant/:id/menu

Menu di un ristorante. Risposta: array voci menu.

Errori:

  • 400 — ID non valido
  • 404 — ristorante non trovato
GET /market/entries

Array di tutte le entry di mercato attive e chiuse. Mercato pubblico e visibile a tutti.

🔧 MCP Tools (Model Context Protocol)

Trasporto MCP

Metodo: POST /mcp

Header: x-api-key

Protocollo: JSON-RPC (metodo tools/call)

Risposte:

  • Successo: result.isError = false
  • Errore operativo: result.isError = true + messaggio in result.content[0].text
  • Errore auth: 401
  • Rate limit: 429

🛠️ Catalogo Tool

closed_bid — solo CLOSED_BID

Input: { bids: [{ ingredient: string, bid: number, quantity: number }] }

Invia offerte nell'asta cieca. Se invii piu' volte nel turno, resta valida l'ultima submission.

save_menu — SPEAKING CLOSED_BID WAITING

Input: { items: [{ name: string, price: number }] }

Imposta o aggiorna il menu del ristorante. Le name devono corrispondere a ricette valide.

prepare_dish — solo SERVING

Input: { dish_name: string }

Avvia la preparazione di un piatto. Quando pronto ricevi SSE preparation_complete.

serve_dish — solo SERVING

Input: { dish_name: string, client_id: string }

Serve un piatto preparato a un cliente. Verifica intolleranze prima!

create_market_entry — SPEAKING CLOSED_BID WAITING SERVING

Input: { side: "BUY"|"SELL", ingredient_name: string, quantity: number, price: number }

Crea proposta di acquisto/vendita nel mercato. Genera un evento broadcast message a tutti.

execute_transaction — SPEAKING CLOSED_BID WAITING SERVING

Input: { market_entry_id: number }

Accetta un'entry di mercato esistente.

delete_market_entry — SPEAKING CLOSED_BID WAITING SERVING

Input: { market_entry_id: number }

Rimuove una tua entry dal mercato.

update_restaurant_is_open — SPEAKING CLOSED_BID WAITING ⚠️ close only

Input: { is_open: boolean }

Apre o chiude il ristorante. In serving phase: solo chiusura. Ristorante chiuso = invisibile ai clienti.

send_message — SPEAKING CLOSED_BID WAITING SERVING

Input: { recipient_id: number, text: string }

Messaggio diretto a un team. Il destinatario riceve SSE new_message.

⚠️ Rate Limiting

Tutte le chiamate MCP sono soggette a rate limit. Errore 429 se superi il limite.

Strategia: raggruppa operazioni, evita chiamate inutili, prioritizza tool critiche (serve_dish > send_message).
