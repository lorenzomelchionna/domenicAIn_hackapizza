  • save_menu
  • prepare_dish / serve_dish

📅 Timeline Evento

Sabato 12:00-14:00: Setup e studio traccia (server attivo, no run)
↓
Sabato 14:00-17:00: Run di Testing (risultati non contano, possibili reset)
↓
Sabato 17:00 - Domenica 10:00: Run Ufficiale (risultati contano, no reset, modifiche codice ok)
↓
Domenica 10:00: STOP THE CODING + Reset completo
↓
Domenica 10:00-12:00: 🏆 GOLDEN RUN (solo agenti, no modifiche, peso alto in classifica)

🔄 Ciclo Completo di un Turno

📡 SSE: game_started → Inizio nuovo turno
↓
SPEAKING Fase 1: Diplomazia e Pianificazione
  • ✅ Parlare con altri team via send_message
  • ✅ Stringere alleanze/accordi
  • ✅ Definire strategia del turno
  • ✅ Impostare menu con save_menu
  • ✅ Aprire/chiudere ristorante
  • ✅ Mercato interno (create/execute/delete market entry)
  • ❌ NO asta ingredienti ancora
📡 SSE: game_phase_changed → phase: "closed_bid"
↓
CLOSED BID Fase 2: Asta Cieca Ingredienti
  • ✅ Inviare offerte con closed_bid
  • ✅ Ingredienti limitati, chi offre di più ha priorità
  • ✅ Possibile ricevere solo parte della quantità richiesta
  • ✅ Ultima submission vince (se mandi più volte)
  • ✅ Ancora possibile: save_menu, mercato interno, send_message
  • ❌ NO preparazione/servizio piatti
📡 SSE: game_phase_changed → phase: "waiting"
↓
WAITING Fase 3: Assestamento Strategico
  • ✅ Analizzare ingredienti ottenuti (GET /restaurant/:id)
  • ✅ Aggiornare menu finale con save_menu
  • ✅ Chiudere/rafforzare accordi
  • ✅ Mercato interno per comprare/vendere ingredienti mancanti
  • ✅ Decidere strategia di servizio
  • ❌ NO preparazione/servizio ancora
📡 SSE: game_phase_changed → phase: "serving"
↓
SERVING Fase 4: Servizio ai Clienti
📡 SSE: client_spawned → { clientName, orderText }
  • ✅ Interpretare richiesta cliente
  • ✅ Preparare piatto con prepare_dish
📡 SSE: preparation_complete → { dish }
  • ✅ Servire con serve_dish (dish_name, client_id)
  • ⚠️ Verificare intolleranze cliente!
  • ⚠️ Cliente non servito o servito male = non paga + danno reputazione
  • ✅ Possibile chiudere temporaneamente se sopraffatti
  • ✅ Mercato interno ancora attivo
  • ❌ NO save_menu (bloccato in serving)

⚠️ REGOLA CRITICA: Ingredienti Deperibili

Alla fine del turno (fase stopped), TUTTI gli ingredienti non utilizzati scadono e vengono persi. Pianifica gli acquisti solo per ciò che puoi realmente cucinare!

📡 SSE: game_phase_changed → phase: "stopped"
↓
STOPPED Fine Turno
  • 🗑️ Ingredienti non usati scadono e vengono eliminati
  • 💰 Aggiornamento saldo ristorante
  • ⭐ Aggiornamento reputazione
  • 📊 Calcolo metriche turno
  • ❌ Tutte le operazioni bloccate
  • ✅ Solo lettura dati (GET endpoints)
↓
Nuovo turno inizia... (loop back a game_started)

📡 Sistema Eventi SSE

Connessione

Endpoint: GET /events/:restaurantId

Header: x-api-key

Regole:

  • Solo una connessione SSE attiva per ristorante
  • Puoi connetterti solo al tuo restaurantId
  • Handshake iniziale: data: connected

📨 Catalogo Eventi

game_started (broadcast a tutti)

Payload: {}

Indica l'inizio di un nuovo turno

game_phase_changed (broadcast a tutti)

Payload: { phase: "speaking" | "closed_bid" | "waiting" | "serving" | "stopped" }

Indica il cambio di fase nel turno corrente

client_spawned (solo al ristorante destinatario)

Payload: { clientName: string, orderText: string }

Un cliente è arrivato e ha fatto una richiesta (solo in serving phase)

preparation_complete (solo al ristorante destinatario)

Payload: { dish: string }

La preparazione del piatto è completata, pronto per essere servito

message (broadcast a tutti)

Payload: { sender: string, payload: any }

Messaggio pubblico (es. nuova entry di mercato creata)

new_message (solo al ristorante destinatario)

Payload: { messageId, senderId, senderName, text, datetime }

Messaggio privato ricevuto da altro team via send_message

heartbeat (broadcast a tutti)

Payload: { ts: number } (epoch milliseconds)

Health check della connessione SSE, ignorare nella logica di gioco

game_reset (broadcast a tutti)

Payload: {}

Reset completo del gioco (es. tra run)

🔑 Pattern Implementazione

1. Aprire connessione SSE all'avvio dell'agente

2. Gestire eventi in modo asincrono (non bloccare)

3. Riconnettere automaticamente se connessione cade

4. Ignorare heartbeat nella logica di gioco

5. Su game_phase_changed: triggerare logica specifica della fase

🔧 Tool MCP Disponibili

Endpoint MCP

Metodo: POST /mcp

Header: x-api-key

Protocollo: JSON-RPC (tools/call)

Risposta successo: result.isError = false

Risposta errore: result.isError = true + messaggio in result.content[0].text

🛠️ Tool per Fase

closed_bid

Input: { bids: [{ ingredient, bid, quantity }] }

Fase: solo closed_bid

Note: Se invii più volte, vale ultima submission

save_menu

Input: { items: [{ name, price }] }

Fase: speaking, closed_bid, waiting

Note: Imposta/aggiorna menu del ristorante

prepare_dish

Input: { dish_name: string }

Fase: solo serving

Note: Avvia preparazione, ricevi event preparation_complete quando pronto

serve_dish

Input: { dish_name: string, client_id: string }

Fase: solo serving

Note: Serve piatto a cliente, verifica intolleranze prima!

create_market_entry

Input: { side: "BUY"|"SELL", ingredient_name, quantity, price }

Fase: speaking, closed_bid, waiting, serving

Note: Crea offerta mercato, broadcast evento message a tutti

execute_transaction

Input: { market_entry_id: number }

Fase: speaking, closed_bid, waiting, serving

Note: Accetta entry di mercato esistente

delete_market_entry

Input: { market_entry_id: number }

Fase: speaking, closed_bid, waiting, serving

Note: Rimuove tua entry di mercato

update_restaurant_is_open

Input: { is_open: boolean }

Fase: speaking, closed_bid, waiting (apri/chiudi), serving (solo chiudi)

Note: Apre o chiude ristorante, ristorante chiuso = invisibile ai clienti

send_message

Input: { recipient_id: number, text: string }

Fase: speaking, closed_bid, waiting, serving

Note: Messaggio diretto a team, ricevitore ottiene evento new_message

⚠️ Rate Limiting

Tutte le chiamate MCP sono soggette a rate limiting. Errore 429 se superi il limite.

Strategia: Raggruppa operazioni, pianifica bene, evita chiamate inutili.

🏗️ Componenti Principali

1️⃣ SSE Listener

  • Connessione persistente a /events/:restaurantId
  • Gestione asincrona eventi
  • Riconnessione automatica
  • Event handler per ciascun tipo evento

2️⃣ State Manager

  • Stato corrente ristorante (saldo, inventario, menu)
  • Fase corrente del turno
  • Storico turni e performance
  • Clienti in attesa/serviti

3️⃣ Strategy Engine

  • Decisioni di bidding (closed_bid)
  • Ottimizzazione menu e pricing
  • Gestione mercato interno
  • Long-term planning

4️⃣ MCP Tool Orchestrator

  • Chiamate MCP via datapizza-ai
  • Rate limiting management
  • Error handling e retry
  • Tool call validation per fase

5️⃣ LLM Agent (datapizza-ai)

  • OpenAILikeClient con Regolo.ai
  • Tool calling per decisioni complesse
  • Prompt engineering per strategie
  • Memory management

6️⃣ Data Layer

  • GET /recipes (ricette disponibili)
  • GET /restaurants (altri ristoranti)
  • GET /bid_history (storico aste)
  • GET /market/entries (mercato interno)

🔄 Loop Principale

1. Inizializzazione
  • Caricare configurazione (API key, restaurant ID)
  • Inizializzare client datapizza-ai
  • Aprire connessione SSE
  • Caricare dati statici (ricette)
↓
2. Event Loop (asincrono)
  • Ascoltare eventi SSE in background
  • Dispatchare eventi a handler specifici
  • Mantenere stato sincronizzato
↓
3. Decision Loop (per fase)
  • speaking: Negoziare, definire strategia turno, impostare menu
  • closed_bid: Analizzare fabbisogni, calcolare bid ottimali, inviare offerte
  • waiting: Verificare ingredienti ottenuti, aggiustare menu, mercato interno
  • serving: Gestire code clienti, preparare/servire piatti, monitorare intolleranze

🎯 Pattern Chiave

Long-term Planning

Durante Run Ufficiale (no reset):

  • Accumulare dati su: pattern di bidding altri team, clientela effettiva, performance piatti
  • Adattare strategie in base a storico
  • Bilanciare esplorazione (nuove ricette) vs exploitation (ricette che funzionano)

Memory Management

Serve memoria persistente tra turni:

  • Storico performance per ricetta
  • Profilo altri ristoranti (aggressività bidding, menu tipico)
  • Accordi/alleanze con altri team
  • Pattern di domanda clienti

Rate Limiting Strategy

Ottimizzare chiamate MCP (429 = troppo frequenti):

  • Raggruppare decisioni (es. menu completo, non una ricetta alla volta)
  • Cache locale stato ristorante
  • Prioritizzare tool call critiche (serve_dish > send_message)

⚠️ Golden Run Requirements

L'agente deve funzionare completamente autonomo:

  • ✅ Gestione errori robusta (retry, fallback)
  • ✅ Riconnessione SSE automatica
  • ✅ Strategie conservative quando info incompleta
  • ✅ No human in the loop
  • ✅ Logging dettagliato per debug post-mortem
